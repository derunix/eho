# Eho Dataset Pipeline

Инструмент для подготовки датасета по книгам Макса Фрая и ручной работы с ним.

Проект умеет:
- извлекать реплики, безмолвную речь и внутренние монологи Макса;
- собирать факты о мире Ехо, персонажах, местах, магии, истории и событиях;
- строить per-book и глобальную базу знаний;
- генерировать синтетические примеры в голосе Макса;
- давать веб-интерфейс для ревью, правки и экспорта итогового датасета.

## Что внутри

- [extract_dialogues.py](extract_dialogues.py) — основной пайплайн extraction.
- [extract_regex.py](extract_regex.py) — быстрый extractor голоса без LLM.
- [dataset_studio.py](dataset_studio.py) — локальная веб-студия для работы с датасетом.
- [dataset_studio_assets](dataset_studio_assets) — фронтенд студии.
- [diagnose_llm.py](diagnose_llm.py) — вспомогательная диагностика модели.
- [tests](tests) — unit, smoke и headless UI tests.

## Что умеет пайплайн

- Поддерживает `.fb2`, `.fb2.zip` и `.txt`.
- Делает semantic chunking по структуре книги.
- Извлекает голос Макса через `regex` или LLM.
- Извлекает знания о мире через многоэтапный LLM-пайплайн.
- Канонизирует сущности внутри книги и между книгами.
- Ведёт глобальную базу знаний и timeline-артефакты.
- Пишет промежуточные checkpoint-файлы и поддерживает `resume`.
- Корректно переживает `Ctrl+C` и ошибки отдельных чанков.
- Сохраняет LLM trace, metadata и историю событий запуска.

## Архитектура пайплайна

Пайплайн разбит на несколько слоёв, чтобы отдельно управлять:
- полнотой извлечения;
- устойчивостью к битым ответам модели;
- качеством fact extraction;
- последующей ручной ревизией.

Высокоуровневый поток такой:

1. Загрузка книги и определение режима обработки.
2. Разбор `fb2`/`txt` в чистый текст.
3. Semantic chunking.
4. Извлечение голоса Макса.
5. Извлечение фактов о мире.
6. Нормализация, канонизация и дедупликация фактов внутри книги.
7. Запись per-book артефактов.
8. Построение глобальной базы знаний по всем книгам.
9. Offline timeline resolution.
10. Ревью и правка через `Dataset Studio`.
11. Экспорт финального датасета.

## Логика принятия решений

### 1. Определение режима книги

По имени файла определяется:
- рассказчик;
- режим обработки.

Режимы:
- `full` — голос Макса + знания + синтетика;
- `voice_only` — только голос;
- `knowledge_only` — только знания + синтетика;
- `knowledge_raw` — только знания, без синтетики.

Это позволяет не смешивать книги, где рассказчик не Макс, с книгами, где важен именно его голос.

### 2. Chunking

Нарезка делается не просто по длине, а по структуре:
- `section`
- смысловые разрывы
- абзацы

Если фрагмент действительно длинный и логически цельный, он может пробить целевой лимит и сохраниться как один chunk. Это сделано сознательно: целостный кусок текста полезнее для extraction, чем механически порезанный кусок без контекста.

### 3. Извлечение голоса

Для голоса есть два режима:
- `regex` — основной быстрый путь для прямой речи, безмолвной речи и монологов;
- `llm` — fallback/альтернатива для более мягкого извлечения.

По умолчанию голос выгоднее брать `regex`, потому что это:
- быстрее;
- стабильнее;
- лучше воспроизводится;
- меньше зависит от капризов модели.

LLM-путь для голоса оставлен как опция и как страховка на сложных случаях.

### 4. Извлечение знаний

Knowledge extraction сейчас многоэтапный.

Сначала идут отдельные треки:
- `world`
- `scene`

`world` ищет:
- устойчивые свойства мира;
- роли и идентичности персонажей;
- правила, институты, именованные места, устойчивые предметы;
- долговременные состояния.

`scene` ищет:
- события;
- изменения состояния;
- решения, открытия, переходы;
- сюжетные факты, которые важны за пределами одного предложения.

Дальше применяется:
- line-based protocol вместо больших JSON-массивов;
- rule-based validation;
- LLM validation на автономность факта;
- retrieval linking к уже накопленной knowledge base;
- канонизация subject;
- дедупликация без потери противоречащих и эволюционирующих фактов.

### 4.1. Трёхэтапная оценка фактов

После того как модель предложила кандидаты фактов, они проходят три разных уровня оценки.

#### Этап 1. Жёсткая структурная и смысловая валидация в коде

На этом этапе без участия LLM отсекается всё, что почти наверняка является мусором:
- битые элементы и неправильная схема;
- placeholder-subject;
- мета-описания вроде `персонаж, который...`;
- тавтологии;
- сценический шум;
- факты, не заземлённые в `PRIMARY CHUNK`;
- конфликт `subject ↔ fact`, когда subject и само утверждение говорят о разных сущностях.

Это самый быстрый и самый дешёвый слой. Его задача:
- резко уменьшить мусор до дорогих этапов;
- не тратить дополнительные вызовы LLM на заведомо плохие кандидаты.

#### Этап 2. Узкая LLM-валидация на автономность

После rule-based фильтра surviving facts отправляются в отдельный validator-prompt.

Validator видит:
- короткий excerpt из `PRIMARY CHUNK`;
- список уже нормализованных кандидатов.

Он отвечает только по сути:
- `keep`
- `drop`

Смысл этого шага — ответить на вопрос:

“Является ли это самостоятельным фактом для базы знаний, который имеет смысл вне одной сцены?”

На этом этапе режутся случаи вроде:
- локальной реакции персонажа;
- сценического пересказа;
- фактов типа `X удивился`, `Y спросил`, `в тексте упоминается`, если они не имеют самостоятельной ценности.

#### Этап 3. Retrieval-linking и оценка относительно уже накопленной базы

Даже хороший факт ещё не обязан становиться новой записью “как есть”.

На третьем этапе кандидат сравнивается с уже накопленной knowledge base:
- ищутся релевантные кандидаты по subject и содержанию;
- решается, не относится ли факт к уже известной сущности;
- subject может быть переиспользован или канонизирован;
- подтверждённые дубли могут быть отброшены;
- новые факты добавляются как новые.

Задача этого слоя:
- не плодить новые subject для уже известных сущностей;
- не терять при этом реально новые состояния и новые факты;
- поддерживать глобальную связность knowledge base между книгами.

Важно: этот этап не должен агрессивно “схлопывать всё похожее”.  
Он специально сделан консервативным, потому что в книгах меняются:
- состояния персонажей;
- отношения;
- правила мира;
- временные фазы одной и той же сущности.

### 5. Две extractor-модели и арбитр

По умолчанию knowledge extraction использует selective ensemble:
- primary extractor;
- secondary extractor;
- arbiter для спорных случаев.

Текущая логика:
- primary модель извлекает факты первой;
- если chunk выглядит подозрительно, запускается secondary;
- совпавшие факты считаются сильными кандидатами;
- несовпавшие отправляются в арбитр;
- арбитр решает `keep / rewrite / drop`.

Ensemble запускается не на каждый “здоровый” chunk, а только на подозрительные случаи, чтобы не убивать скорость на ровном месте.

Подозрительный chunk — это обычно:
- слишком мало surviving facts;
- слишком высокий процент отбраковки;
- format failures;
- пустой или почти пустой результат.

### 6. Почему факты не должны “слишком хорошо” дедуплицироваться

Для мира Фрая важно, что:
- герои меняются;
- отношения меняются;
- правила мира меняются;
- многие книги описывают разные временные состояния одной и той же сущности.

Поэтому система специально старается:
- не склеивать противоречащие факты;
- хранить разные состояния мира;
- учитывать `time_scope`;
- разделять “одна и та же сущность” и “одно и то же утверждение”.

## Промежуточные форматы данных

Проект сознательно пишет много промежуточных файлов. Это нужно для:
- `resume`;
- дебага;
- ручной ревизии;
- воспроизводимости;
- offline-пересборки global KB и timeline.

### Chunks

Файл:
- `chunks_<book>.jsonl`

Одна строка содержит, как минимум:
- `idx`
- `chapter`
- `chunk_text`
- `dialogues`
- `knowledge`

Пример:

```json
{
  "idx": 0,
  "chapter": "Глава 1",
  "chunk_text": "Дом у Моста служит штабом Тайного Сыска...",
  "dialogues": [...],
  "knowledge": [...]
}
```

### Голос

Файл:
- `voice_<book>.jsonl`

Формат совместим с fine-tune chat dataset:

```json
{
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "Контекст или реплика собеседника"},
    {"role": "assistant", "content": "Реплика Макса"}
  ]
}
```

### Knowledge

Потоковый файл:
- `knowledge_<book>.jsonl`

Финальный per-book snapshot:
- `knowledge_<book>.json`

Базовый формат факта:

```json
{
  "category": "place",
  "subject": "Дом у Моста",
  "fact": "Дом у Моста служит штабом Тайного Сыска.",
  "time_scope": "timeless",
  "source_book": "1. Лабиринты Ехо/1. Чужак.fb2",
  "chapter": "Глава 1",
  "chunk_idx": 0
}
```

Ключевые поля:
- `category` — `character`, `place`, `magic`, `history`, `event`, `creature`, `custom`
- `subject` — каноническая сущность
- `fact` — автономное утверждение
- `time_scope` — временная рамка
- `source_book`, `chapter`, `chunk_idx` — трассировка происхождения

### Синтетика

Файлы:
- `synth_<book>.jsonl`
- `synth_progress_<book>.jsonl`

Синтетика хранится в том же `messages`-формате, что и `voice`.

### Global knowledge

Файлы:
- `knowledge_base_raw.json`
- `knowledge_base_raw.txt`
- `knowledge_base.json`
- `knowledge_base.txt`

`raw` — просто сборка по всем per-book знаниям.  
Финальная база — после глобальной канонизации и дедупликации.

### Timeline

Файлы:
- `timeline_resolution_raw.json`
- `timeline_graph.json`
- `timeline_graph.txt`

`timeline_resolution_raw.json` — результат offline-LLM resolution по группам фактов и чанков.  
`timeline_graph.json` — агрегированный граф сущностей, событий и связей.

### Trace и metadata

Файлы:
- `metadata.json`
- `metadata_history.jsonl`
- `llm_traces/`
- `editor_workspace/llm_jobs/`

`metadata.json` — текущий snapshot состояния.  
`metadata_history.jsonl` — append-only журнал событий запуска.  
`llm_traces/` — полные request/response traces всех вызовов LLM.

## Time scope и timeline resolution

`time_scope` уже закладывает важную временную структуру:
- `past`
- `current`
- `ended`
- `timeless`
- `unclear`

Этого достаточно, чтобы:
- не терять смену состояний;
- отделять исторические факты от текущих;
- позже строить timeline subsets для обучения.

После extraction запускается отдельный offline-этап:
- читает факты и исходные chunks;
- строит локальные событийные группы;
- собирает timeline graph по сущностям, событиям и отношениям.

Это отдельный слой поверх extraction, а не попытка “заставить модель сразу понять всю хронологию”.

## Dataset Studio

Студия запускается локально и работает поверх папки `output/`.

Запуск:

```bash
python dataset_studio.py --output-dir ./output
```

По умолчанию адрес:

```text
http://127.0.0.1:8766
```

Что есть в интерфейсе:
- `Facts` — просмотр, оценка, правка, удаление и создание фактов.
- `Voice` и `Synth` — правка обучающих примеров.
- `Themes` — создание, объединение и редактирование тем.
- `LLM trace` — просмотр прошлых запусков, rerun trace и ручной новый запуск.
- `Chunks` — исходные куски текста, связанные факты и примеры, создание факта из чанка.
- `Pipeline` — snapshot из `metadata.json`, история событий и `llm_jobs`.
- `Timeline` — просмотр timeline-графа, если он уже построен.

Все пользовательские правки сохраняются отдельно в `output/editor_workspace/` и не затирают исходные pipeline-файлы. После ревью можно экспортировать итоговый датасет для дообучения.

## Как работать в веб-студии

Рекомендуемый практический цикл такой:

1. Прогнать extraction и дождаться появления `output/`.
2. Открыть `Dataset Studio`.
3. Просмотреть `Facts` и отсортировать шум/полезные факты.
4. Использовать `Chunks`, чтобы видеть исходный текст и создавать недостающие факты.
5. Использовать `Themes`, чтобы группировать факты по сущностям, сюжетам или подсетам датасета.
6. Проверить `Voice` и `Synth` и отредактировать плохие примеры.
7. Смотреть `LLM trace`, если нужен дебаг конкретного ответа модели.
8. Смотреть `Pipeline`, если нужно понять, на каком этапе что-то пошло не так.
9. Смотреть `Timeline`, если уже построен граф событий и связей.
10. Экспортировать финальный датасет.

Почему студия устроена именно так:
- ручная правка нужна не только для facts, но и для samples;
- источник ошибки почти всегда понятнее через chunk и LLM trace;
- undo обязателен, потому что ревью — это серия итераций, а не одноразовая правка;
- все операции пишутся отдельно, чтобы исходные pipeline-артефакты оставались воспроизводимыми.

## Быстрый старт

1. Создай виртуальное окружение.
2. Установи зависимости:

```bash
pip install -r requirements.txt
```

3. Положи книги в папку `books/`.
4. Запусти локальную модель, например:

```bash
ollama run gemma4:latest
```

5. Запусти extraction:

```bash
python extract_dialogues.py
```

Пример с явными параметрами:

```bash
python extract_dialogues.py --workers 2 --voice-extractor regex --chunk-size 2500
```

Серверный preset для 48 GB GPU с лучшим балансом скорость/качество по бенчмарку `2026-04-16`:

```bash
python extract_dialogues.py --llm-preset server-48gb-balanced --workers 1
```

Серверный preset для мощной видеокарты с упором на большие модели и quality:

```bash
python extract_dialogues.py --llm-preset server-48gb-quality --workers 1
```

Любой preset можно точечно переопределить явными флагами, например:

```bash
python extract_dialogues.py --llm-preset server-48gb-balanced --knowledge-validate-model gemma4:31b
```

Короткие CLI-алиасы для knowledge-ролей тоже поддерживаются:
- `--primary-model` = `--knowledge-extract-model`
- `--secondary-model` = `--knowledge-extract-model-secondary`
- `--arbiter-model` = `--knowledge-arbiter-model`
- `--validator-model` = `--knowledge-validate-model`

Пример ручной server-конфигурации без preset:

```bash
python extract_dialogues.py \
  --primary-model gemma4:26b \
  --secondary-model gemma4:31b \
  --arbiter-model gemma4:26b \
  --validator-model gemma4:26b \
  --workers 1 \
  --no-resume
```

Если не нужен автозапуск `ollama serve`:

```bash
python extract_dialogues.py --no-auto-serve
```

Если нужен полный пересчёт без resume:

```bash
python extract_dialogues.py --no-resume
```

## Почему используются именно такие модели

Текущий default stack не взят “по бенчмарку на бумаге”, а по реальным прогонам на книгах Макса Фрая.

Сейчас в проекте есть два уровня конфигурации:
- локальный default для бытового железа;
- серверные presets для 48 GB GPU и больших моделей.

### Primary extractor: `gemma4`

Причины:
- лучшая устойчивость на русском материале именно в текущем extraction pipeline;
- меньше мусорных summary/entity-catalog ответов;
- лучше держит формат line-based extraction;
- в реальных тестах давала лучший баланс качества и устойчивости.

Минус:
- не самая быстрая.

### Secondary extractor: `qwen3:8b`

Причины:
- полезна как “второе мнение” на сложных чанках;
- иногда подхватывает факты, которые primary пропускает;
- ошибки у неё не полностью совпадают с `gemma4`.

Почему не основная:
- на реальных бенчмарках как single-model extractor вела себя хуже;
- чаще ломала формат;
- давала больше локального шума.

### Arbiter / validator / linker

По умолчанию логика заточена под то, что арбитраж и валидация работают той же более надёжной моделью, что и primary extractor, если не задан явный override.

Причина простая:
- для арбитра важнее дисциплина и консервативность, чем recall.

## Локальные бенчмарки и почему выбран текущий стек

В репозитории не лежат сами benchmark outputs, но локально они прогонялись на диагностических кейсах из `Чужака`:
- биография Макса;
- Кимпа как именованный персонаж и его роль;
- `Обжора Бунба` как именованное место;
- сложный чанк про Шурфа, перчатки и социальное правило мира.

Сравнивались:
- `gemma4`
- `phi4-mini`
- `qwen2.5:7b`
- `qwen3:4b`
- несколько mixed-model схем с арбитром

Характерные результаты одного из основных раундов:

| Конфигурация | Hit rate | Hits | Facts | Parse errors | Avg sec/case |
| --- | ---: | ---: | ---: | ---: | ---: |
| `gemma4` | 0.600 | 3/5 | 49 | 4 | 97.37 |
| `phi4-mini` | 0.000 | 0/5 | 0 | 18 | 18.80 |
| `qwen2.5:7b` | 0.000 | 0/5 | 0 | 24 | 47.13 |

Отдельные mixed-model раунды:

| Конфигурация | Hit rate | Hits | Facts | Parse errors | Avg sec/case |
| --- | ---: | ---: | ---: | ---: | ---: |
| `gemma4 + gemma4` | 0.600 | 3/5 | 47 | 6 | 93.03 |
| `gemma4 + phi4-mini` | 0.600 | 3/5 | 50 | 8 | 129.74 |
| `gemma4 + qwen3:4b` | 0.600 | 3/5 | 49 | 6 | 127.44 |
| поздний раунд `gemma4 + gemma4` | 0.800 | 4/5 | 52 | 0 | 138.93 |

Практический вывод из этих запусков:
- простая замена primary-модели на `qwen` или `phi` ухудшала extraction;
- лучший single-model baseline — `gemma4`;
- secondary extractor нужен не всегда, а именно для сложных чанков;
- ensemble полезнее как selective fallback, чем как обязательный full-pass на каждый chunk;
- line-based extraction оказался заметно надёжнее больших JSON-массивов.

Именно поэтому текущий подход такой:
- `gemma4` как primary;
- `qwen3:8b` как selective secondary;
- line-protocol;
- rule-based + LLM validation;
- explicit arbiter на спорных фактах.

## Серверные presets после бенчмарка 2026-04-16

Для мощных GPU в `extract_dialogues.py` добавлены готовые presets.

### `server-48gb-balanced`

Использует:
- base / primary: `mistral:7b`
- secondary extractor: `gemma4:26b`
- arbiter / validator / linker: `gemma4:26b`

Зачем:
- это практический mixed-stack из свежего бенчмарка;
- `mistral:7b` дал лучший Quality×Speed;
- `gemma4:26b` дал лучший quality среди реально полезных single-model конфигураций;
- selective ensemble даёт более сильный recall без полного перехода на медленный large-model-only режим.

### `server-48gb-quality`

Использует:
- base / primary: `gemma4:26b`
- secondary extractor: `qwen3-coder:30b`
- arbiter / validator / linker: `gemma4:26b`

Зачем:
- режим для мощной карты, где можно позволить себе large-model second opinion;
- `gemma4:26b` остаётся главным quality-anchor;
- `qwen3-coder:30b` добавляет ещё одно сильное мнение на сложных чанках.

Это не “самый быстрый” путь, но это лучший из найденных локально компромиссов между:
- качеством фактов;
- контролем мусора;
- воспроизводимостью;
- скоростью на бытовом локальном железе.
